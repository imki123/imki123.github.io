2.귀여운 React Native 앱 Catbook!
2020-10-03
안냥하세요 임기입니다 :D 오늘은 React Native로 만든 Catbook을 소개하려고 글을 썼어요. Catbook은 귀여운 고양이와 강아지 이미지를 볼 수 있는 간단한 네이티브 앱이랍니다. 외부 사이트의 api 정보를 가져오는걸 가공해서 보여준답니다. 간단한 구글 번역도 제공해요 ㅎㅎ 아래는 web앱 주소와 Play store 주소에요~ https://imki123.github.io/catbook/ https://play.google.com/store/apps/details?id=com.github.imki123 그런데 React Native가 뭐냐고요? React Native는 facebook에서 제작한 프론트엔드 라이브러리로 React와 유사한 방식으로 모바일 native앱을 만드는 프로그램이랍니다. React 코드를 작성할 줄 알면 아주 쉽게 RN 앱도 만들 수 있어요 RN의 가장 큰 장점은 크로스플랫폼을 지원한다는 겁니다. 하나의 코드로 안드로이드, ios, web 앱을 동시에 만들수 있어요! 정말 놀랍죠? RN를 사용하지 않으면 똑같이 동작하는 프로그램을 만들기 위해서 세가지 언어로 세번 코딩을 해야하는 끔찍한 일이 벌어진답니다. RN는 이 외에도 정말 많은 장점이 있답니다 :D RN에는 정말 편리한 기능인 hot-reloading이 있어요. 네이티브 앱의 경우 코드를 수정하면 코드를 새로 컴파일하고 그걸 에뮬레이터에 올리고 테스팅 하는데 시간이 꽤 걸려요. 그래서 개발을 하다보면 흐름이 끊겨 답답하고 컴파일 하는데만 수 시간을 소모하곤 한답니다. 하지만 hot-reloading은 state를 초기화하지 않고 수정한 코드를 바로 화면에 적용시켜주어서 테스팅 시간을 획기적으로 줄일 수 있답니다. live-reloading은 소스코드 수정 시 앱 전체를 새로고침 하는건데요. 설정을 통해서 선택해주시면 됩니다. 그리고 RN을 얘기하다보면 빼놓을 수없는게 있는데요. 바로 expo입니다. RN을 처음 세팅하려면 꽤나 어렵고 복잡한 설정들을 잡아줘야합니다. 웹개발자에게 이런 세팅은 상당히 번거로운 일이죠. 이를 간편하게 세팅해주는게 바로 expo입니다. expo는 이외에도 아주 강력한 기능을 지원해줍니다. 바로 expo app을 통해 내 스마트폰에서 실시간으로 디버깅을 할 수 있는건데요. 정말 놀라운건 윈도우 환경에서 만든 앱을 바로 아이폰에서 테스팅 할 수 있습니다.. 언빌리버블.. 정말 expo 팀에게 찬사를 아끼지 않을 수 없네요. 또 아주 훌륭한 기능이 있는데요. 바로 expo publish입니다. 네이티브 앱은 구글플레이나 앱스토어에 앱을 게시해야 다른 사람들이 앱을 설치하고 사용할 수 있는데요. 앱을 게시하고 적용되는데 꽤 시간이 소요됩니다. 길게는 일주일 이상 수정사항을 적용하지 못하는 경우가 생길 수 도 있습니다. 자그마한 수정사항이라도 내가 원할 때에 앱을 게시 할 수 없다는건 아주 큰 리스크가 될 수 있죠. 하지만 expo는 실시간 publishing을 지원합니다! 내가 앱을 수정하고 expo publish를 하면 수정사항이 곧바로 앱에 적용됩니다. 정말 놀랍지 않을 수가 없습니다... 여기서 끝이 아닙니다. expo는 RN에서 다루지 않는 다양한 기능들을 지원합니다. 예를 들어 gps의 경우 RN에서 제공하는 기능은 몇 가지 뿐이지만 expo는 수 십 가지의 기능을 지원합니다. expo를 활용하시면 개발에 큰 이점을 줄 수 있답니다. 여러분들도 expo를 이용해서 간단한 네이티브 앱을 만들어보시면 어떨까요 ^^?

어몽어스는 미션이 있는 마피아 게임이에요 최대 10명이서 게임을 할수 있고 게임을 시작하면 크루원 vs 임포스터로 진영이 갈려요 크루원은 우주선 내에있는 미션들을 수행하면 왼쪽 위에 미션 게이지가 올라가고 그 미션 게이지가 꽉 차면 크루원의 승리. 임포스터는 크루원이 미션을 모두 끝내기 전에 크루원을 죽여서 임포스터 수와 일치시키면 임포스터의 승리. 임포스터는 킬을 들키지 않기 위해 방해공작을 펼칠 수 있답니다. 위 이미지는 시작 시에 처음 위치해 있는 식당이에요. 식당 한가운데는 Emergency 버튼이 있어서 이걸 누르면 긴급 회의가 시작되요. 긴급 회의에서는 임포스터를 찾아내는 투표를 진행할 수 있어요. 투표는 한 명을 선택하거나, 스킵하거나, 아무 것도 안할 수 있어요. 한 명이 가장 많이 선택되면 그 사람은 회의가 끝나고 우주선 밖으로 추방된답니다 ㅠㅠ 추방된 사람이 임포스터이면 게임은 크루원들의 승리로 끝나고, 임포스터가 아니면 게임은 계속 진행되요. 또 위에 보시면 지도에 노란색 느낌표가 있어요. 이건 크루원들이 수행해야하는 미션을 나타내는데요, 모든 크루원이 미션을 완료하면 크루원의 승리로 끝나게 된답니다. 임포스터에게 죽거나 추방되면 유령이 되는데 유령도 미션을 완료해야 미션이 모두 끝나게 되요. 죽어서도 미션을 게을리하면 안된답니다. ㅎㅎ 위 이미지는 미션이 가능할 때 나타나는 노란색 표시에요. 노란색 근처로 가게되면 오른쪽 아래의 USE 버튼이 활성화되고 이때 USE 버튼을 누르면 미션을 수행할 수 있답니다. 미션의 종류는 맵에 따라서 다양하니까 많이 해보셔야해요 ㅎㅎ 이제 임포스터의 역할에 대해 설명할건데요. 이건 사보타지 버튼을 눌렀을 때 나타나는 화면입니다. 임포스터는 사보타지를 통해 우주선을 고장내거나, 문을 차단하여 몰래 크루원들을 죽여서 승리할 수 있어요. 위의 X가 있는 동그라미 버튼은 해당 방의 문을 닫을 수 있는 버튼이에요. 문은 10초 동안 닫혀 있고, 각각 30초의 쿨타임이 적용된답니다. [원자로] 버튼을 누르면 경고음이 들리면서 30초의 제한시간과 원자로실의 위아래 미션이 활성화 되는데요. 제한시간 안에 원자로 미션을 완료하지 못하면 우주선이 고장나면서 임포스터의 승리로 끝나게 된답니다. 원자로 미션은 두 명이 위 아래로 나뉘어 손바닥 화면을 터치하고 있으면 쉽게 해제된답니다. [산소공급실] 버튼은 원자로와 마찬가지로 30초의 제한시간내에 미션을 완료해야합니다. 산소 공급실 미션은 관리실과 산소실 두군데로 나뉘어 있어서 두군데 모두 미션을 해야합니다. 5자리 비밀번호가 적힌 숫자판을 누르고 OK 버튼을 누르면 된답니다. [통신실] 버튼은 시간제한은 없고 미션이 안보이고 보안실 CCTV와 관리실 지도가 먹통이 된답니다. 미션을 해제하려면 통신실로 가서 동그란 버튼을 돌려야하는데 버튼을 돌리다보면 빨간불이 사라질 때가 있어요. 그 때 버튼을 멈추면 미션 성공. 위 이미지는 임포스터가 벤트를 타는 모습이에요 ㅎㅎ 실제로 보면 쓩하고 들어가는게 되게 귀엽답니다. 벤트를 타면 벤트 속에 숨어있거나 인근 다른 방의 벤트로 나올 수 있습니다. 킬을 하고 벤트로 숨는다든지 벤트에 숨어있다가 나와서 킬을 한다든지 활용을 잘하면 감쪽 같이 킬을 할 수 있답니다. 단 벤트를 탈 수 있는건 임포스터 뿐이니 안들키게 조심해야해요 ㅎㅎ. 참 베트를 타는 도중에는 킬 쿨타임이 줄어들지 않고 사보타지도 걸 수 없으니 꼭 필요할 때만 사용하도록 하세요. 임포스터가 킬을 할 경우 임포스터가 시체 위로 순간이동하여 빨간 피가 튄답니다 ㄷㄷ.. 저 장면은 1초 정도만 볼 수 있어서 잠깐 한 눈 팔다보면 눈앞에서 임포스터가 킬을 해도 못보는 경우도 있답니다. 집중해서 보셔야해요. 죽게 되면 아래처럼 덩그러니 고깃덩이같은 모습을 한 시체가 된답니다... 시체를 발견하면 오른쪽에 REPORT 버튼이 활성화 된답니다. 리포트를 하게되면 [원자로]와 [산소실] 사보타지가 즉시 해제되고 회의가 소집됩니다. 회의에서 어디서 누가 죽었는지 얘기하고 의심가는 임포스터를 찾아내야 하겠죠? 위 이미지는 보안실 CCTV를 켤 때 보이는 화면입니다. 왼쪽 위에서부터 1번 화면은 산소실-항해실-보호막제어실과 연결되는 통로를 보여줍니다. 2번 화면은 식당-관리실-창고를 연결하는 통로입니다. 3번은 상부엔진-의무실-식당을 연결하는 통로랍니다. 4번은 상부엔진-원자로-보안실-하부엔진을 연결하는 통로입니다. 전기가 고장 나도 CCTV는 정상적으로 작동한답니다. 전기실 사보타지가 걸렸다면 CCTV로 임포스터를 감시하는 것도 좋은 방법이에요. 위 이미지는 관리실 지도 화면입니다. 크루원들이 어디에 있는지 한눈에 볼 수 있습니다. 단 색깔은 알 수가 없고 시체 또한 크루원으로 표시되니 오랜시간 움직이지 않는 크루원이 있다면 가서 확인해 보셔야해요 자 여기까지 기초적인 어몽어스 설명을 마쳤어요. 마피아 게임보다 상당히 복잡한 요소가 들어있어서 처음엔 어렵지만 함께 힘을 합쳐 임포스터를 찾다보면 시간 가는 줄 모르고 어몽어스에 푹 빠질거에요 ㅎㅎ 친한 친구들과 하면 두배로 재미있답니다. 다들 즐몽하세요~ 😊

5.GitHub Pages 설정해보기
2020-09-26
안녕하세요 임기입니다 :D 오늘은 GitHub Pages에 대해 알아봅시다. GitHub Pages는 github에서 제공하는 무료 웹호스팅 서비스입니다. 우리가 github에 html파일을 저장하면 인터넷을 통해 웹페이지를 볼 수 있다는 거죠. 그것도 무료로 말이에요. 대단하죠? 매우 쉬운 사용방법을 알려드릴게요. 우선 github에 가입하고 나서 레포지토리를 만들고 Readme.md 파일을 자동 생성하고 Settings에 들어가봅시다. 그러면 스크롤 중간쯤에 GitHub Pages라는 섹션이 있을겁니다. 여기서 Source의 Branch를 선택해주면 설정이 끝납니다. 기본으로 master 브랜치를 설정해주시면 해당 브랜치의 html파일 또는 md파일을 웹 화면에 보여주게됩니다. 이렇게 간단하게 정적 html을 이용해서 웹페이지를 만들 수 있습니다. 대부분의 github.io 블로그들이 해당되죠. 그외에도 동적인 방법을 사용할 수도 있는데요. React 같은 SPA(Single Page Application)가 이에 해당합니다. cors는 어렵다. cors는 Cross-Origin-Resource-Sharing의 준말이다. 즉, 서로 다른 출처 간의 자원 공유를 차단하기 위한 정책이다. 어떤 웹페이지에서 다른 도메인의 자원을 가져오고자 할때, 도메인이 다를 경우 이를 차단하는 정책이다. 나같은 경우는 프론트에서 백엔드 정보를 가져와야하는데 둘의 도메인이 달라서 설정해줬다. cors는 두군데에서 설정해야한다. 백엔드, 프론트엔드 둘다이다. 백엔드는  1. 서버프레임워크에 cors 정책을 설정한다. const app = Koa() app.cors({ origin: 'https:kiyoung.com',  credentials: true,  exposeHeaders: ['page-count'] }) 등을 설정해야한다. 2. 서버의 https 프로토콜을 설정한다. app.proxy = true koa의 경우 proxy를 설정해줘야 https 프로토콜이 동작한다. 3. 쿠키를 set 하기위해서는 서버의 cookieOptions도 설정해줘야한다. const cookieOptions = { secure: true, //https 프로토콜 sameSite: 'none' } 로컬 환경에서 테스트 할때는 http 프로토콜이므로 secure를 false로 바꿔주어야한다. 안하면 error가 발생한다.. 로컬 환경에서는 ctx.request.protocol을 조회해서 간단한 메서드를 만들어 미들웨어에 추가해주자. 프론트엔드는 데이터를 요청할 때 credentials 설정을 포함하여 전송해야한다. 1. axios의 경우 axios.post(url, { withCredentials: true }) 를 설정해야한다. 1. fetch의 경우 fetch(url,{ method: 'POST', mode: 'cors', credentials: 'include' }) p.s) axios 라이브러리로 cookie-set을 할 경우 잘 안된다. 아무래도 axios의 오류인 것으로 생각된다. 기본 자바스크립트 라이브러리인 fetch는 잘 작동한다. 그래서 대부분 axios를 사용하고 ctx.cookie나 ctx.state를 조작해야 하는 경우는 fetch를 사용하고 있다. 둘 다 사용하다보니 둘 다 까먹지 않아서 좋은 것 같기도 하다. axios를 써서 편하게 하는 것도 좋지만 fetch도 사용할 줄 알아야 하겠지?
5.블로그 프론트엔드에 useContext() 적용기
2020-09-21
안녕하세요? 블로그를 제작하면서 Redux 등의 상태관리 라이브러리를 사용하지 않다가 오늘 useContext()를 적용해봤습니다. 리액트는 컴포넌트에서 상태값들을 관리 할 수 있는데, Top-down 방식으로 상위 컴포넌트의 상태를 하위 컴포넌트에게 전달해줄 수 있습니다. 문제는 상태값을 전달하기 위해 수 많은 컴포넌트를 거쳐야하는 props drilling이 매우 빈번하게 발생할 수 있습니다. 이를 보완해주는 것이 Context API였고 useContext() Hook으로 발전하면서 함수형 컴포넌트에서도 손쉽게 사용할 수 있게 되었습니다. 사용법은 꽤나 간단합니다. 루트 컴포넌트에 store를 만들어서 전역상태로 관리하고 싶은 상태값들을 객체 형태로 담습니다. // App.js const [login, setLogin] = useState(false) const [user, setUser] = useState({ username: 'imki123' }) const sayHello = () => { console.log('Hello!') } //전역상태값들을 store에 집어 넣습니다. 네이밍은 편한대로 사용하시면 됩니다 :D const store = { login, setLogin, user, setUser, sayHello, } 그 다음 App 컴포넌트 위쪽에 AppContext = React.createContext()를 생성하고 아래쪽에서 export { AppContext } 합니다. 다음으로는 <AppContext.Provider>로 App을 감싸주고 value에 store를 담아 하위 컴포넌트들에게 전달해줍니다. // App.js import React, { useState } from 'react' const AppContext = React.createContext() function App() { const [login, setLogin] = useState(false) // ... const store = { login, setLogin, user, setUser, sayHello, } // ... return ( <AppContext.Provider value={store}> <div id="app"> <Second> <Third> <Fourth/> </Third> <Second/> </div> </AppContext.Provider> } export { AppContext } export default App 그리고 AppContext의 하위 컴포넌트에 가서 AppContext를 import하고 store = React.useContext(Appcontext)를 불러오면 props drilling 없이 store 객체를 사용할 수 있습니다. // Fourth.js import React from 'react' import { AppContext } from './App' function Fourth() { const store = React.useContext(AppContext) const logout = e => { store.setLogin(false) store.setUser(false) } return( {login && <div> <div>{store.user.username}님 환영합니다 :D</div> <button onClick={logout}>로그아웃</button> </div>} ) } export default Fourth 어때요 쉽죠? :D 세션 정보같이 여러 곳에서 사용해야 하는 전역 상태는 store로 관리하고 간단한 상태들은 props로 전달하면 훨씬 깔끔한 앱을 만드는데 도움이 될거에요~
